# 任务管理说明

## 当前模型

当前版本的任务系统采用“后端主导、前端显示”的模型：

- 后端队列是唯一真实状态来源
- 前端页面切换、刷新、重新进入课程页都不会影响已入队任务
- 页面恢复时会从 `/tasks` 重新拉取状态

## 为什么默认串行

默认启用：

```bash
SERIAL_TASK_EXECUTION=1
```

原因很直接：

- 下载、转录、生成笔记会竞争磁盘、显卡和 API 配额
- 对这个项目来说，串行比并发更可控
- 用户在课程 A 和课程 B 间切换时，不需要担心显卡同时跑多份转录

只有显式设置 `SERIAL_TASK_EXECUTION=0` 时，`WORKER_COUNT` 才会生效。

## 任务入队规则

- 任务提交后直接进入后端队列
- 重复任务允许继续排队
- 不在前端做“静默吞并”

这样做是为了保持行为可预期：

- 用户点了两次，就是两个任务
- 系统不会擅自改写任务语义

## 任务执行规则

任务真正开始执行时，服务端会重新检查当前文件状态：

- 如果前序任务已经完成了所需产物，就自动跳过
- 如果前序任务只做了一部分，就继续补齐剩余步骤
- 只有用户明确确认覆盖时，才会重做已有产物

这套逻辑的目标不是“自动合并任务”，而是“在串行前提下尽量少做无意义重复工作”。

## 资源锁

同一课节共用一把资源锁，用来避免并发写同一组输出文件：

- 音频
- 视频
- 字幕
- 笔记

即使以后放开并发，同一课节的文件写入仍然不会同时进行。

## 前端面板恢复

`sjtu_video_helper.js` 的职责现在主要是：

- 提交任务
- 轮询任务状态
- 页面恢复时从服务端拉取任务

它不再作为任务真相来源。

## 当前已修复的问题

- 下载视频会真实走视频分支，不再落到音频下载
- 重新下载、重新转录、重新生成的覆盖语义已统一
- 轮询不会重复注册
- 顶层浅层错误面板会被主动清理

## 仍需注意

- 任务状态目前仍保存在内存里，重启服务后历史任务无法恢复
- 如果未来需要更强恢复能力，建议落到 SQLite
