# 任务管理功能使用指南

## 🎯 新增功能概览

### 1. 任务持久化 + 面板强制恢复（页面刷新不丢失）

-   **问题**：之前刷新页面或切换课程后，所有任务状态都会丢失
-   **解决**：任务状态保存在后端服务器，页面加载时自动从服务器恢复所有任务；即便此时还没检测到视频，面板也会被强制创建，任务立即可见
-   **效果**：即使关闭浏览器再打开，正在进行的任务依然可见并继续更新

### 2. 任务取消功能

-   **操作**：每个任务右侧有一个红色的 `✕` 按钮
-   **效果**：
    -   正在排队/处理中的任务：点击后标记为"已取消"，worker 线程会跳过执行
    -   已完成/出错的任务：按钮变为 `🗑`，点击从列表中删除

### 3. 批量清空已完成任务

-   **操作**：点击面板中的 `🗑️ 清空已完成` 按钮
-   **效果**：一键清空所有已完成、已出错、已取消的任务，只保留正在进行的任务
-   **提示**：清空操作会强制刷新任务列表，确保前后端状态一致

### 4. 任务元数据持久化

-   **内容**：后台为每个任务记录 `课程名 / 课节标题 / 操作类型` 等元信息
-   **效果**：前端刷新后仍能显示“生成 AI 笔记 · 2025-10-17 08:55 程序语言...”这样的任务描述，不再丢失上下文

## 🔧 技术实现

### 后端改造（`auto_study_server.py`）

#### 新增 API 端点

1. **GET `/tasks`**

    - 返回所有任务列表（按创建时间倒序）
    - 用于页面加载时恢复任务状态

2. **DELETE `/tasks/<task_id>`**

    - 取消/删除指定任务
    - 正在进行的任务：标记为 `cancelled`
    - 已完成的任务：从列表中删除

3. **POST `/tasks/clear`**
    - 批量清空已完成/出错/取消的任务
    - 只保留 `queued`、`processing`、`downloading` 状态的任务

#### 任务取消机制

```python
cancelled_tasks = set()  # 存储已取消的任务ID

def is_task_cancelled(task_id):
    """检查任务是否已被取消"""
    with task_lock:
        return task_id in cancelled_tasks

def worker():
    while True:
        task = task_queue.get()
        task_id = task['id']

        # 在执行前检查是否已被取消
        if is_task_cancelled(task_id):
            logger.info(f"[Task {task_id}] 已被取消，跳过执行")
            update_task(task_id, "cancelled", 0, "任务已取消")
            task_queue.task_done()
            continue

        # 正常执行任务...
```

### 前端改造（`sjtu_video_helper.js`）

#### 新增函数

1. **`restoreTasksFromServer()`**

    - 页面加载时调用（1 秒后自动执行）
    - 从 `/tasks` API 获取所有任务
    - 恢复任务 UI 并继续轮询正在进行的任务

2. **`cancelTask(taskId)`**

    - 调用 `DELETE /tasks/<task_id>` 取消任务
    - UI 更新由轮询自动完成

3. **`deleteTask(taskId)`**

    - 删除已完成的任务
    - 同时从 UI 和服务器移除

4. **`clearFinishedTasks()`**
    - 调用 `POST /tasks/clear`
    - 调用成功后自动强制刷新任务列表，UI 与后端保持一致

#### UI 优化

-   每个任务添加取消/删除按钮
-   任务状态颜色区分：
    -   🟢 绿色：已完成
    -   🔴 红色：出错
    -   ⚪ 灰色：已取消
    -   🔵 蓝色：进行中

## ⚠️ 注意事项

### 1. 网络超时问题

你遇到的 `[WinError 10060]` 错误是 **网络连接超时**，与代码无关。

**排查步骤**：

-   检查是否能访问 `https://generativelanguage.googleapis.com`
-   如果在国内，需要配置代理：
    ```python
    # 在 core_processor.py 的 __init__ 中添加
    import os
    os.environ["HTTPS_PROXY"] = "http://127.0.0.1:7890"  # 替换为你的代理地址
    ```
-   或者使用 VPN 确保网络畅通

### 1.5 笔记生成不完整/截断问题

**问题表现**：生成的笔记只有部分内容（如只有逐字稿），缺少后续部分（如"关键时间节点"、"结构化笔记"等）。

**原因**：

-   **输出长度限制**：Gemini API 有 `max_output_tokens` 限制，默认可能只有 2048-8192 tokens，对于长课程（>40 分钟）容易触发截断
-   **音频过长**：超过 1 小时的课程可能需要更大的输出容量

**解决方案（已自动实现）**：

-   ✅ 已将 `max_output_tokens` 增加到 `16384`（约 12000 汉字）
-   ✅ 增加了自动截断检测：如果生成的笔记缺少必要部分，会自动重试最多 2 次
-   ✅ 如果重试后仍不完整，会在笔记末尾添加警告信息，提示缺失的部分

**用户操作**：

-   如果看到笔记末尾有 **⚠️ 生成警告**，说明笔记不完整
-   可以尝试取消当前任务并重新生成（系统会自动重试）
-   对于超长课程（>1.5 小时），建议人工分段处理

> **仍需注意**：服务端任务信息目前存放在内存中，如果重启 `auto_study_server.py`，历史任务无法恢复。未来可考虑持久化到 SQLite/JSON 文件。

### 2. 任务状态同步

-   任务状态通过**每秒轮询**更新（`setInterval(..., 1000)`）
-   如果网络不稳定，可能会有 1-2 秒的延迟

### 3. 服务器重启

-   如果重启 `auto_study_server.py`，所有任务状态会丢失
-   建议使用 `screen` 或 `pm2` 保持服务长期运行

### 4. 并发数量

-   默认 4 个并发 worker（`WORKER_COUNT=4`）
-   如果 Gemini API 有速率限制，可以降低：
    ```bash
    $env:WORKER_COUNT=2
    python -m auto_study_server
    ```

## 📝 API 文档速查

| 端点           | 方法   | 功能             | 返回                                 |
| -------------- | ------ | ---------------- | ------------------------------------ |
| `/process`     | POST   | 生成 AI 笔记     | `{task_id, message}`                 |
| `/download`    | POST   | 下载音频/视频    | `{task_id, message}`                 |
| `/tasks`       | GET    | 获取所有任务     | `{task_id: {status, progress, ...}}` |
| `/tasks/<id>`  | GET    | 获取单个任务状态 | `{status, progress, message}`        |
| `/tasks/<id>`  | DELETE | 取消/删除任务    | `{status, message}`                  |
| `/tasks/clear` | POST   | 清空已完成任务   | `{removed_count}`                    |
